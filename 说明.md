# 后端说明

Updated: 2026-04-09

这份文档是给“第一次看这个后端、觉得层次有点乱”的人准备的。

如果你只想先抓住大方向，可以先记一句话：

`platform-contracts` 定义“大家说的话长什么样”，`platform-core` 负责“真正干活”，`platform-api` 负责“把外面的请求接进来，再把底层能力接出去”。

---

## 1. 先看总目录

`backend` 现在只有 3 个正式模块：

```text
backend
├─ platform-contracts
├─ platform-core
└─ platform-api
```

可以这样理解：

- `platform-contracts`
  - 公共协议层
  - 放各种 `Record / DTO / Enum`
  - 前端、后端、持久化层都要认识这些类型
- `platform-core`
  - 核心业务层
  - 真正的聊天、研究、记忆、图执行、工具调用，都在这里
- `platform-api`
  - 接口与装配层
  - Controller、Spring 配置、数据库适配、Redis、RocketMQ、Spring AI 接线都在这里

如果把它类比成一个餐厅：

- `contracts` 是菜单和点菜单
- `core` 是后厨
- `api` 是前台、收银、传菜和水电煤

## 3. 每个模块到底干嘛

## `platform-contracts`

这个模块最适合用一句话理解：

“这里只放公共名词，不放真正干活的逻辑。”

比如这些目录：

- `contracts/message`
  - 聊天消息、运行事件、research draft 这些对象
- `contracts/task`
  - 任务相关的数据结构
- `contracts/research`
  - 研究报告、来源、引用、findings、iterations
- `contracts/memory`
  - 短期记忆、长期记忆、profile、facts
- `contracts/workspace` / `contracts/thread`
  - 工作区和线程的基础结构
- `contracts/model` / `contracts/skill` / `contracts/search`
  - 模型配置、技能配置、Web 搜索配置
- `contracts/artifact` / `contracts/document`
  - 上传产物和文档记录

什么时候看它：

- 想搞清楚接口返回了什么字段
- 想看前后端共享的数据长什么样
- 想知道某个 `Record` 到底属于哪个领域

一句人话总结：

- 这里回答的是“系统里有哪些数据对象”
- 不是“这些对象怎么被算出来”

## `platform-core`

这是后端真正的核心。

一句话：

“业务脑子基本都在这里。”

### `platform-core/agent/core`

这是最值得先看的地方。

#### `application`

放“流程型应用服务”。

这里常见的是：

- `ConversationMemoryService`
  - 负责组装线程级短期记忆视图
- `ShortTermMemoryProjectionService`
  - 把消息、任务、草案投影成 `ThreadMemoryView`
- `LongTermMemoryJobScheduler`
  - 判断什么时候该做长期记忆抽取
- `LongTermMemoryExtractionService`
  - 真正调用模型做长期记忆提炼

人话：

- 这是“把几块零件串起来”的那层
- 很多“一个动作要走好几步”的逻辑会在这里

#### `chat`

普通聊天相关逻辑。

比如：

- 路由聊天
- 选择是否用工具
- 组织普通对话执行

人话：

- 用户发一条普通消息后，系统怎么回，主要看这里

#### `interaction`

这是聊天入口更核心的一层。

最关键的类之一就是：

- `InteractionGraphNodeService`

它对应 interaction graph 的各个节点。

人话：

- 用户发消息后，要先读记忆、再决定走普通聊天还是 deep research 起草、再决定要不要用工具
- 这种“有分叉、有状态”的执行过程，主要在这里落地

#### `research`

深度研究相关逻辑。

通常可以分成两段看：

- `research/scoping`
  - 先把用户模糊需求变成 draft
- `research/execution`
  - 真正跑后台研究任务

这里很关键的类有：

- `ResearchExecutionGraphNodeService`
  - 研究任务图的节点实现

人话：

- 如果你在查“deep research 为什么这么走”，这里就是主战场

#### `shared`

放聊天和研究都会共用的东西。

比如：

- prompt 组装时共用的格式化器
- memory context 格式化器

人话：

- 公共零件库

### `platform-core/graph`

这里是“图执行内核”。

重点类：

- `GraphRuntimeFactory`
  - 最核心的运行入口之一
- `CheckpointConfiguration`
  - 图状态怎么做 checkpoint
- `InteractionState`
  - 聊天/起草流程的状态对象
- `ResearchTaskState`
  - 研究任务流程的状态对象

人话：

- 这里不是业务细节
- 这里是“这个项目怎么把业务流程跑成状态机/图”的地方

如果你觉得项目有点 LangGraph4j 味道太重，看的就是这里。

### `platform-core/runtime`

这里主要是抽象接口和运行时通用能力。

常见内容会包括：

- repository 接口
- dispatcher 接口
- publisher 接口

人话：

- 这是“核心业务想依赖，但不想直接绑死具体实现”的那层
- 你可以理解为 core 眼中的“端口”

### `platform-core/memory`

文档与检索相关底层能力。

比如：

- `DocumentStore`
- `ChunkIndexStore`
- `ContextAssembler`

人话：

- 上传的文档怎么变成可检索内容，主要是这层在兜底

### `platform-core/tools`

工具调用相关的核心逻辑。

人话：

- 模型要不要调用工具
- 工具长什么样
- 工具怎么受控

都和这里有关

### `platform-core/workspace`

工作区目录、文件、artifact 等底层能力。

人话：

- 这是“文件落在哪里、工作区路径怎么组织”的地方

## `platform-api`

这个模块的关键词是：

“对外接口 + Spring 装配 + 基础设施实现。”

### `controller`

最容易看懂的一层。

主要就是接口入口：

- `MessageController`
  - 聊天 SSE 入口
- `ResearchController`
  - draft 获取、更新、启动
- `TaskController`
  - 任务详情、取消、研究结果读取
- `MemoryController`
  - 线程记忆、长期记忆、profile、facts
- `UploadController`
  - 文件上传、分片上传
- `WorkspaceController` / `ThreadController`
  - 工作区、线程管理
- `ArtifactController` / `DocumentController`
  - 产物与文档列表
- `ModelSettingsController`
  - 模型配置
- `SkillController`
  - 技能配置
- `WebSearchSettingsController`
  - Web 搜索配置

人话：

- 想知道“某个接口进来后先去哪里”，先看 controller

### `config`

Spring 装配层。

重点文件：

- `GraphConfig`
  - interaction graph、research graph、AgentExecutionService 等核心 bean 的装配点
- `AiConfig`
  - 模型调用相关配置
- `PersistenceConfig`
  - 持久化相关装配
- `MemoryConfig`
  - 短期/长期记忆链路装配
- `AsyncConfig`
  - 本地异步 / RocketMQ
- `ToolingConfig`
  - 工具体系配置
- `UploadConfig`
  - 上传配置
- `PlatformProperties`
  - 所有 `platform.*` 配置项的大本营

人话：

- 这层像“总配电箱”
- 真正干活的代码不一定在这里，但所有线大多从这里接起来

### `ai`

Spring AI 适配层。

比如：

- provider client resolver
- prompt service
- turn execution support

人话：

- 这里回答的是“模型到底怎么被调用”
- 不是“业务为什么要这么调用”

### `persistence`

数据库实现层。

这里通常又会看到 `mybatisplus` 下面分成：

- `entity`
  - 数据库表对应实体
- `mapper`
  - MyBatis Mapper
- `repository`
  - 面向 core 暴露的仓储实现
- `convertor`
  - 领域对象和数据库对象互转
- `typehandler`
  - 特殊字段类型处理

人话：

- 这层是“数据库怎么存、怎么查”
- 如果你在追一个字段为什么进库后长这样，来这里看

### `service`

API 层补充服务。

当前最值得关注的是：

- `TaskResearchReadService`

它负责把研究任务的输出读出来，统一对外提供：

- report
- plan
- iterations
- findings
- sources
- citations

人话：

- 这是“给前端读研究结果”的整理层

### `upload`

上传相关基础设施。

里面会有：

- 分片上传 session
- 上传状态存储
- Redis/InMemory 状态实现

人话：

- 大文件、断点、分片这类事，主要看这里

### `messaging`

RocketMQ 相关适配。

人话：

- 如果系统切到 `platform.async.mode=rocketmq`
- 任务和事件怎么进消息队列，看这里

### `cache`

Redis 缓存适配。

人话：

- 主要是“让它更快”
- 不是“让它更准”

### `skill` / `search` / `model`

这些是用户配置与能力接线层：

- `skill`
  - 技能配置、密钥存储
- `search`
  - Web 搜索配置
- `model`
  - 模型提供商配置

人话：

- 这一层更像“用户可调设置”的后端实现

### `src/main/resources`

主要有三块：

- `db/migration`
  - Flyway SQL 迁移
- `mapper`
  - MyBatis XML
- `application.properties`
  - 默认配置

人话：

- Java 代码之外的重要运行资料都在这里

## 4. 你应该按什么顺序理解这个后端

我建议按这个顺序看：

### 第一层：先看入口

看这些：

- `backend/platform-api/src/main/java/com/xg/platform/PlatformApiApplication.java`
- `backend/platform-api/src/main/java/com/xg/platform/api/controller`

目的：

- 搞清楚外部有哪些接口

### 第二层：看总装配

重点看：

- `GraphConfig.java`
- `PlatformProperties.java`

目的：

- 搞清楚系统是怎么接起来的

### 第三层：看主执行骨架

重点看：

- `AgentExecutionService.java`
- `GraphRuntimeFactory.java`
- `InteractionGraphNodeService.java`
- `ResearchExecutionGraphNodeService.java`

目的：

- 搞清楚消息和研究任务到底怎么跑

### 第四层：按专题往下钻

想查什么就进什么专题：

- 查记忆：`agent/core/application` + `memory` + `api/cache`
- 查上传：`api/upload` + `workspace`
- 查数据库：`api/persistence/mybatisplus`
- 查研究结果：`api/service/TaskResearchReadService`

## 5. 一句话版地图

如果你后面容易忘，我给你压缩成最短版本：

- `contracts`
  - 定义数据长什么样
- `core`
  - 定义业务怎么跑
- `api`
  - 定义请求怎么进来、配置怎么接上、数据库怎么落地

再压缩一层：

- `controller` 看入口
- `config` 看接线
- `core` 看主逻辑
- `persistence` 看进库
- `graph` 看流程骨架

## 6. 现在这个项目“乱”主要乱在哪

说人话，主要不是模块数量乱，而是这几种东西混在一起看时会让人晕：

1. 有一层“业务逻辑”
2. 有一层“图节点逻辑”
3. 有一层“Spring 装配”
4. 有一层“数据库实现”

它们都在做“同一件事”的不同切面，所以第一次看会觉得绕。

最实用的办法不是试图一次看懂全部，而是先问自己：

- 我现在是在看接口入口？
- 还是在看业务流程？
- 还是在看图状态机？
- 还是在看数据库落地？

一旦先把这个问题答清楚，目录就会顺很多。
